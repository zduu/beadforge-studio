# BeadForge Studio 改进优化计划

## 目标

把当前项目从“功能可用”推进到“稳定、可验证、适合持续扩展”。优先降低 3MF 多色解析、模型切片和多层导出流程的回归风险，同时改善真实制作拼豆图纸时的工作效率。

## 当前判断

- 项目已能构建通过，基础图片转图纸、模型上传、3D 预览、模型切片、编辑和导出流程已经具备。
- 主要复杂度集中在 `src/App.tsx` 和 `src/lib/modelToLayered.ts`。
- 当前缺少自动化测试，后续继续扩展 3MF 兼容性和切片规则时风险较高。
- `PROJECT_PLAN.md` 中有部分计划已经完成，需要后续同步状态，避免重复规划。

## 优先级总览

| 优先级 | 方向 | 目标 |
| --- | --- | --- |
| P0 | 自动化测试 | 防止 3MF 多色解析、图纸转换和切片逻辑回归 |
| P0 | 模型模块拆分 | 降低 `modelToLayered.ts` 继续扩展的维护成本 |
| P1 | 切片诊断 | 让用户理解模型缩放、层数和空层原因 |
| P1 | 多层项目导入 | 让导出的 layered JSON 可以重新编辑 |
| P2 | 批量导出 | 支持真实拼装流程的一键层图包 |
| P2 | 性能与体积 | 优化大模型处理体验和构建产物提醒 |
| P3 | 文档同步 | 让 README、PROJECT_PLAN 和实际功能保持一致 |

## P0：补自动化测试

### 问题

项目当前没有测试脚本。`3MF` 解析和模型切片属于高风险逻辑，靠手动验证容易漏掉颜色丢失、耗材槽合并、层数异常和背景统计错误。

### 建议任务

1. 引入 `Vitest`。
2. 增加 `npm test` 和 `npm run test:watch` 脚本。
3. 为纯函数优先补测试：
   - `src/lib/color.ts`
   - `src/lib/pattern.ts`
   - `src/lib/layeredPattern.ts`
4. 给 `modelToLayered.ts` 拆分后的核心逻辑补测试：
   - 3MF filament color 保留
   - Bambu `paint_color` 映射
   - STL 默认颜色兜底
   - 层数计算
   - 简单几何切片结果
5. 建立测试样本目录，样本文件不进入生产构建。

### 验收标准

- `npm test` 可以在本地一次通过。
- 多色 3MF 不会退化成单色。
- `3MF Filament` 独立色号不会被强制合并到 PLA Basic。
- 单层图纸的背景统计、颜色替换、裁剪和撤销相关数据可验证。

## P0：拆分模型处理模块

### 问题

`src/lib/modelToLayered.ts` 同时负责文件识别、STL 解析、3MF 解包、XML 解析、Bambu 颜色映射、几何变换、切片和预览序列化。继续在同一个文件里扩展会导致测试和维护成本快速上升。

### 建议拆分

```text
src/lib/model/
  fileType.ts
  stlParser.ts
  threeMfArchive.ts
  threeMfParser.ts
  bambuColors.ts
  geometry.ts
  slicer.ts
  preview.ts
  index.ts
```

### 实施策略

1. 先移动纯函数，不改外部行为。
2. 保持 `modelFileToPreviewData` 和 `modelFileToLayeredPattern` 作为稳定公共入口。
3. 每拆出一个模块就补对应测试。
4. 最后让原 `modelToLayered.ts` 只保留兼容导出，或改成聚合入口。

### 验收标准

- 拆分后 `npm run build` 通过。
- 公共 API 不变，`modelWorker` 不需要大规模改动。
- 解析、颜色、几何和切片可以分别测试。

## P1：增加切片诊断信息

### 问题

用户看到切片结果异常时，目前很难判断是模型方向、格距、层高、目标层数、缩放比例还是模型几何本身导致。

### 建议任务

1. 在切片结果中保存诊断信息：
   - 原始模型尺寸
   - 旋转后尺寸
   - 缩放比例
   - 缩放后尺寸
   - 目标层数
   - 实际层数
   - 每层占用格数
   - 空层数量
2. 在界面上增加“切片信息”区域。
3. 当模型过大、层数过高或网格过密时提前提示。
4. 对空层、极少格层和超高层数给出明确状态文案。

### 验收标准

- 用户不用读源码也能理解模型为什么被缩放成当前尺寸。
- 空层、层数异常和模型过小能在 UI 中被解释。
- 诊断信息能随 layered JSON 一起导出。

## P1：支持多层项目重新导入

### 问题

当前项目已经支持导出多层 JSON，但编辑流程更需要能够重新导入并继续查看、编辑、导出。

### 建议任务

1. 增加 `validateLayeredPattern`。
2. 扩展项目导入逻辑，识别：
   - `single-layer`
   - `layered-model`
3. 导入多层项目后恢复：
   - 图纸宽高
   - 色卡
   - 全部层数据
   - 当前层预览
   - 模型源信息
   - 切片参数
   - 模型方向
4. 导入失败时显示明确错误。

### 验收标准

- 导出的 layered JSON 可以原样导入。
- 导入后颜色统计与导出前一致。
- 导入后可以继续切换层、旋转当前层、导出统计。

## P2：批量导出制作包

### 问题

真实拼装多层模型时，单独导出 JSON 不够直接。用户通常需要每层图、每层 CSV、总统计和颜色表。

### 建议任务

1. 增加 ZIP 导出依赖或使用浏览器原生能力实现打包。
2. 支持一键导出：
   - 每层 PNG
   - 每层 CSV
   - 总颜色统计 CSV
   - 每层颜色统计 CSV
   - 颜色映射 JSON
   - 项目 JSON
3. 文件名包含模型名、层号和尺寸。
4. 大层数导出时显示处理中状态。

### 验收标准

- 一个多层项目可以导出完整制作包。
- 导出的统计数据与 UI 可见统计一致。
- 文件命名可读，适合直接整理和打印。

## P2：性能和构建体积优化

### 问题

构建通过，但 Three.js 相关 chunk 有体积提醒。大模型处理也需要持续关注浏览器响应性。

### 建议任务

1. 确认 3D 预览组件只在需要时加载。
2. 分析 `OrbitControls` 和 Three.js chunk 是否还能进一步拆分。
3. 大模型切片前增加成本估算：
   - 三角面数量
   - 网格面积
   - 目标层数
   - 预计总格数
4. 对整体 3D 预览继续做抽样和上限保护。
5. 为 Worker 增加更细进度阶段：
   - 读取
   - 解压
   - 解析
   - 归一化
   - 切片
   - 生成预览

### 验收标准

- 常见模型处理期间 UI 不明显卡死。
- 大任务开始前用户能看到成本提示。
- 构建提醒被解释、降低或明确接受。

## 风险控制

- 不在同一次改动中同时重构模型模块和改变切片算法。
- 先补测试，再改复杂解析和切片逻辑。
- 样本模型只放测试或手动验证目录，不进入生产包。
- 保持导出的 JSON 向后兼容，新增字段必须可选。
- 对用户已有项目文件保持宽松读取，严格写出。
